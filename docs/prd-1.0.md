# PRD 1.0

작성일: 2026-06-30  
수정일: 2026-07-02

## 1. 제품 개요

- 팀 이름: 원더독스
- 제품 유형: 경력단절 여성 특화 유연근무 채용 플랫폼
- 핵심 고객: 유연근무 인재가 필요한 중소기업·스타트업 채용담당자 또는 실무 팀장
- 핵심 사용자: 경력 복귀를 원하지만 근무 시간·거리 제약이 있는 경력단절 여성
- 핵심 방향: 기업이 유연근무 공고를 우선 등록하면, 지역·근무조건·직무경험을 기준으로 가까운 경력단절 여성 인재와 연결하고, 실제 업무 기반 과제로 핵심인재를 선별하게 한다.

## 2. 문제 정의

중소기업과 스타트업은 풀타임 1명을 채용하기에는 업무량과 비용이 부담스럽지만, 즉시 실무를 맡길 수 있는 경력자는 필요하다. 특히 유연근무, 시간제, 재택·혼합근무처럼 다양한 조건을 수용할 수 있는 인재를 찾을 채널이 부족하다.

경력단절 여성은 실무 경험과 역량이 있어도 육아, 돌봄, 거주지, 출퇴근 시간 제약 때문에 일반 채용 플랫폼의 풀타임 중심 공고에 지원하기 어렵다. 가까운 지역에서 유연하게 일할 수 있는 기업 정보를 찾기 어렵고, 경력 공백 때문에 역량을 증명할 기회도 부족하다.

그 결과 기업은 적합한 사람을 찾지 못해 채용을 미루고, 경력단절 여성은 다시 일할 수 있는 조건의 일자리를 발견하지 못한다.

## 3. 제품 목표

- 기업이 유연근무 가능 공고를 쉽게 등록하게 한다.
- 경력단절 여성이 거주지와 가까운 유연근무 기회를 찾게 한다.
- 기업과 지원자를 지역, 근무조건, 직무경험 기준으로 우선 매칭한다.
- 실제 업무 기반 과제를 통해 경력 공백보다 현재 역량을 중심으로 평가하게 한다.
- 채용담당자가 과제 결과를 비교해 핵심인재를 빠르게 선별하게 한다.

## 4. 핵심 가치

원더독스는 유연근무가 가능한 기업과 경력 복귀를 원하는 경력단절 여성을 가까운 지역 안에서 연결하고, 실제 업무 과제를 통해 현재 실무 역량을 검증하는 채용 플랫폼이다.

## 5. 사용자 정의

### 기업 사용자

- 50인 이하 중소기업·스타트업의 대표, 채용담당자, 실무 팀장
- 풀타임 채용은 부담스럽지만 실무 경험이 있는 인재가 필요한 사람
- 유연근무, 시간제, 재택·혼합근무 조건을 제시할 수 있는 기업
- 지원자의 실제 업무 수행 능력을 보고 채용하고 싶은 기업

### 경력단절 여성 사용자

- 출산, 육아, 돌봄, 이사 등으로 경력이 중단된 여성
- 기존 경력과 실무 역량을 바탕으로 다시 일하고 싶은 사람
- 집 또는 생활권 근처의 일자리를 선호하는 사람
- 풀타임보다 유연근무, 시간제, 재택·혼합근무를 선호하는 사람

## 6. MVP 범위

### 포함

- 기업 유연근무 조건 포함 공고 등록
- 기업 공고 내용을 기반으로 한 사전 과제 AI 생성
- 구직자 유연근무 조건 필터링 검색
- 재택근무 가능 여부 필터
- 일 8시간 미만 근무 가능 여부 필터
- 지도 기반 내 주변 공고 보기

### 제외

- 경력단절 여성 프로필 정식 등록
- 지원서 제출과 지원 현황 관리
- 과제 제출
- 과제 결과 비교 및 후보자 선별
- 추천 인재 목록
- 면접 또는 제안 요청
- 급여 정산, 계약서 작성, 전자서명
- 장기 근태 관리
- 자동 합격·불합격 결정
- 외부 채용 사이트 연동
- 정교한 AI 평가 점수 자동 산정

## 7. 핵심 기능

### 7.1 유연근무 조건 포함 공고 등록

- 기업은 공고 등록 시 유연근무 조건을 필수로 입력한다.
- 입력 항목은 공고 제목, 직무, 근무지역, 근무형태, 근무시간, 일 8시간 미만 가능 여부, 재택 가능 여부, 주당 근무일, 필요 직무역량, 예상 보수 범위를 포함한다.
- 공고는 경력단절 여성이 조건을 빠르게 판단할 수 있게 카드 형태로 보여준다.

### 7.2 사전 과제 AI 생성

- 기업이 공고의 직무, 필요 역량, 실제 업무 내용을 입력하면 AI가 사전 과제 초안을 생성한다.
- 생성 결과에는 과제 제목, 과제 설명, 제출 형식, 예상 소요 시간, 평가 기준이 포함된다.
- 기업은 AI가 생성한 과제를 수정한 뒤 공고에 연결한다.
- 과제는 짧고 명확해야 하며 무상 노동으로 오해될 만큼 큰 업무를 요구하지 않는다.

### 7.3 유연근무 조건 필터링 검색

- 구직자는 공고 목록에서 유연근무 조건을 기준으로 검색한다.
- 필터는 재택근무 가능 여부, 일 8시간 미만 근무 가능 여부, 근무지역, 직무를 포함한다.
- 지도 화면에서 내 주변 공고만 볼 수 있다.
- 공고 카드에는 근무지역, 재택 가능 여부, 일 8시간 미만 가능 여부, 근무시간이 명확히 표시된다.

## 8. 주요 사용자 흐름

### 기업 흐름

1. 기업이 회원 유형을 채용담당자로 선택한다.
2. 기업 정보와 근무지역을 등록한다.
3. 유연근무 조건을 포함해 공고를 등록한다.
4. 실제 업무 내용을 입력한다.
5. AI가 생성한 사전 과제 초안을 확인하고 수정한다.
6. 공고와 과제를 함께 등록한다.

### 지원자 흐름

1. 지원자가 회원 유형을 지원자로 선택한다.
2. 공고 목록에서 유연근무 조건을 필터링한다.
3. 재택근무 가능 공고를 확인한다.
4. 일 8시간 미만 근무 가능 공고를 확인한다.
5. 지도에서 내 주변 공고를 확인한다.
6. 관심 공고의 상세 내용을 확인한다.

## 10. 매칭 기준 초안

- MVP에서는 자동 인재 매칭보다 공고 검색 필터링을 우선 구현한다.
- 지역 기준: 공고 근무지역과 구직자의 현재 위치 또는 선택 지역의 근접성
- 근무조건 기준: 재택근무 가능 여부, 일 8시간 미만 가능 여부, 근무시간
- 지도 기준: 내 주변 공고만 보기에서 가까운 공고를 우선 노출
- 후속 버전에서 지원자 프로필 기반 추천과 과제 결과 기반 후보자 비교를 검토한다.

## 11. 정책 및 주의사항

- 과제는 실제 채용 검증 목적이어야 하며, 기업이 제출물을 무상 실무 결과물로 사용하는 구조는 지양한다.
- 경력단절 사유를 필수 입력으로 요구하지 않는다.
- 지원자의 상세 주소는 요구하지 않고 위치 기반 검색에 필요한 범위만 사용한다.
- 기업은 유연근무 조건을 공고에 명확히 표시해야 한다.
- AI 생성 과제는 기업이 최종 검토하고 수정할 수 있어야 한다.
- 평가 기준은 구직자에게도 이해 가능한 수준으로 공개한다.

## 13. 협업 규약 결정 요약

### 팀 구성

- 팀장: 정나래
- 팀장은 PR 최종 병합과 배포를 담당한다.

### AI 작동 규칙

- AI는 요청받은 코드와 문서 수정, 실행 확인, 빌드 확인, 린트 확인, 커밋, PR 생성을 수행한다.
- 파일 삭제, 대규모 이동, 의존성 추가, 공통 구조 변경은 먼저 확인받는다.
- 유지해야 할 공통 구조는 SPA와 역할 기반 화면 분기다.
- 문제 정의서와 유저 플로우 설계서는 수정하지 않고 참조만 한다.

### 협업 방식

- 작업은 역할 화면, 공통 구조, 데이터/Firebase, 문서 수정 단위로 나눈다.
- 개인별 작업 브랜치에서 작업하고 `main`은 배포 가능한 상태로 유지한다.
- 브랜치 이름은 `student/이름/작업명` 형식을 사용한다.
- 모든 기능 브랜치는 `main` 병합 전 PR을 만든다.
- AI가 PR을 작성하고 리뷰하며, 팀장이 최종 승인한다.
- PR은 하나씩 순서대로 병합한다.
- 작업 단위는 독립 구현을 기본으로 하고, 중복 구현을 감수한다.
- 데이터 이름 사전은 공유한다.

### 디자인 기준

- 참고 방향: 위시켓처럼 신뢰감 있는 매칭 플랫폼
- 팀 디자인 방향: 원더독스 브랜드에 맞는 부드럽고 친근한 업무 도구 UI
- 메인 컬러: 보라/라벤더 계열
- 보조 색상: 흰색, 밝은 라벤더, 옅은 회색, 진한 회색, 중간 회색
- 선호 요소: 둥근 카드, 밝은 라벤더 배경, 상태 배지, 지원자 비교 표, 친절한 안내 문구
- 비선호 요소: 진한 보라색 과다 사용, 강한 그림자, 복잡한 장식, 너무 작은 글씨
- 약한 디자인 불일치는 후속 정리 가능하다.
- 공통 구조를 깨는 변경은 후속으로 미루지 않는다.

### 데이터 기준

- 역할 값은 `candidate`, `recruiter`, `interviewer`를 사용한다.
- 필드명은 영어 `camelCase`를 사용한다.
- 상태값은 자유 텍스트가 아니라 정해진 값만 사용한다.
- 새 컬렉션, 필드, 상태값, 역할 값은 구현 전에 `docs/data.md` 공통 이름 사전에 먼저 추가한다.
- 화면 컴포넌트와 데이터 접근 코드는 분리한다.
- Firebase 연결 코드는 한 곳에서 관리한다.

### Firebase 기준

- Firebase 설정값은 `.env`에 저장한다.
- `.env`는 Git에 올리지 않는다.
- `.env.example`에는 값 없이 변수 이름만 공유한다.
- 지금은 Hosting 중심으로 사용한다.
- Firestore와 Storage는 필요 시 사용한다.
- 배포 담당은 팀장이다.
- 실제 사용자 데이터가 들어가기 전 Firestore 보안 규칙을 검토한다.

## 15. 비밀값 관리

Firebase Web SDK 설정값은 이 문서에 기록하지 않는다. 실제 값은 `.env`에만 저장하고, 공유용 변수 이름은 `.env.example`에만 둔다.
