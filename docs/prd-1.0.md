# PRD 1.0

작성일: 2026-06-30

## 1. 제품 개요

- 팀 이름: 원더독스
- 제품 유형: 유연근무 경력자 채용 플랫폼
- 1번 고객: 50인 이하 규모 중소기업·스타트업의 실무 팀장
- 핵심 방향: 유연근무로 일할 검증된 경력자를 찾고, 실제 업무 기반 사전 과제로 역량을 확인한 뒤 채용할 수 있게 한다.

## 2. 문제 정의

풀타임 1명은 과한 업무를 가진 중소기업 실무 팀장은, 경력자에게 유연근무로 맡겨 인건비를 줄이고 오래 함께 갈 사람을 뽑고 싶다. 하지만 그런 조건에 맞는 경력자를 찾을 채널이 부족하고, 찾더라도 지금 실제로 일을 할 수 있는지 검증하기 어렵다.

그 결과 채용을 미루거나 풀타임 정규직만 고집하게 되고, 팀은 과부하에 걸린다. 외주로 해결해도 품질과 소통 문제로 다시 일을 떠안는 상황이 생긴다.

## 3. 핵심 가치

유연근무 경력자를 어디서 찾을지 모르는 중소기업 팀장의 고통을, 유연근무로 일할 검증된 경력자 채널과 실제 업무 기반 사전 과제로 해결한다.

## 4. 핵심 기능

- 유연근무 조건 공고와 경력자 지원 풀
- 회사 실제 업무 입력을 바탕으로 한 사전 과제 생성
- 지원자 과제 제출과 결과물 뷰어
- 지원자 결과물 비교 뷰

## 5. 첫 사용 시나리오

회사가 유연근무 조건으로 공고를 올리고 채용할 직무의 실제 업무를 입력한다. 시스템은 이를 사전 과제로 변환한다. 지원자는 과제를 풀어 제출하고, 회사는 결과물을 나란히 비교해 채용 여부를 판단한다.

## 6. 화면 흐름 초안

### 화면 1. 유연근무 경력자 채널

- 가치: 유연근무 경력자를 만날 채널
- 행동: 회사가 유연근무 조건으로 공고를 올린다.
- 기능: 유연근무 조건 공고, 경력자 지원 풀
- 주요 구성: 경력자 풀 보기, 유연근무 공고 올리기, 이용 가이드 보기
- 다음 행동: 제안 보내기 또는 관심 인재 추가

### 화면 2. 사전 과제 생성

- 가치: 진짜 할 수 있는지 검증
- 행동: 회사가 실제 업무를 입력하면 사전 과제로 변환된다.
- 기능: 업무 입력, 질문지 답변, 과제 자동 생성, 수정, 미리보기, 최종 등록
- 다음 행동: 등록 또는 임시저장된 공고에 과제 연결

### 화면 3. 지원자 결과 비교

- 가치: 맞는 사람만 추려 보기
- 행동: 회사가 과제 결과와 조건으로 지원자를 좁힌다.
- 기능: 결과물 비교 뷰, 과제 피드백, 면접 요청, 추가 과제 요청
- 다음 행동: 과제 전형 통과자 대상 면접 여부 확정

## 7. 협업 규약 결정 요약

### 팀 구성

- 팀장: 정나래
- 팀장은 PR 최종 병합과 배포를 담당한다.

### AI 작동 규칙

- AI는 요청받은 코드와 문서 수정, 실행 확인, 빌드 확인, 린트 확인, 커밋, PR 생성을 수행한다.
- 파일 삭제, 대규모 이동, 의존성 추가, 공통 구조 변경은 먼저 확인받는다.
- 유지해야 할 공통 구조는 SPA와 역할 기반 화면 분기다.
- 문제 정의서와 유저 플로우 설계서는 수정하지 않고 참조만 한다.

### 협업 방식

- 작업은 역할 화면, 공통 구조, 데이터/Firebase, 문서 수정 단위로 나눈다.
- 개인별 작업 브랜치에서 작업하고 `main`은 배포 가능한 상태로 유지한다.
- 브랜치 이름은 `student/이름/작업명` 형식을 사용한다.
- 모든 기능 브랜치는 `main` 병합 전 PR을 만든다.
- AI가 PR을 작성하고 리뷰하며, 팀장이 최종 승인한다.
- PR은 하나씩 순서대로 병합한다.
- 작업 단위는 독립 구현을 기본으로 하고, 중복 구현을 감수한다.
- 데이터 이름 사전은 공유한다.

### 디자인 기준

- 참고 방향: 위시켓처럼 신뢰감 있는 매칭 플랫폼
- 팀 디자인 방향: 원더독스 브랜드에 맞는 부드럽고 친근한 업무 도구 UI
- 메인 컬러: 보라/라벤더 계열
- 보조 색상: 흰색, 밝은 라벤더, 옅은 회색, 진한 회색, 중간 회색
- 선호 요소: 둥근 카드, 밝은 라벤더 배경, 상태 배지, 지원자 비교 표, 친절한 안내 문구
- 비선호 요소: 진한 보라색 과다 사용, 강한 그림자, 복잡한 장식, 너무 작은 글씨
- 약한 디자인 불일치는 후속 정리 가능하다.
- 공통 구조를 깨는 변경은 후속으로 미루지 않는다.

### 데이터 기준

- 역할 값은 `candidate`, `recruiter`, `interviewer`를 사용한다.
- 필드명은 영어 `camelCase`를 사용한다.
- 상태값은 자유 텍스트가 아니라 정해진 값만 사용한다.
- 새 컬렉션, 필드, 상태값, 역할 값은 구현 전에 `docs/data-guide.md` 공통 이름 사전에 먼저 추가한다.
- 화면 컴포넌트와 데이터 접근 코드는 분리한다.
- Firebase 연결 코드는 한 곳에서 관리한다.

### Firebase 기준

- Firebase 설정값은 `.env`에 저장한다.
- `.env`는 Git에 올리지 않는다.
- `.env.example`에는 값 없이 변수 이름만 공유한다.
- 지금은 Hosting 중심으로 사용한다.
- Firestore와 Storage는 필요 시 사용한다.
- 배포 담당은 팀장이다.
- 실제 사용자 데이터가 들어가기 전 Firestore 보안 규칙을 검토한다.

## 8. 논의할 항목

- 성사 수수료 또는 첫 달 보수 기준의 구체 비율
- 첫 MVP에서 경력자 풀, 공고 등록, 사전 과제 생성, 결과 비교 중 어디까지 구현할지
- 실제 DB 구조 확정 시점

## 9. 비밀값 관리

Firebase Web SDK 설정값은 이 문서에 기록하지 않는다. 실제 값은 `.env`에만 저장하고, 공유용 변수 이름은 `.env.example`에만 둔다.
