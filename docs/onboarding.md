# 협업 온보딩

처음 레포를 클론한 팀이 협업 기준을 **함께 이해하고**, 필요한 결정만 문서에 반영하기 위한 진행 문서입니다.

## 목적

- 팀원이 문서 구조를 먼저 이해한 뒤 작업을 시작하게 합니다.
- 팀마다 달라질 수 있는 협업 기준을 순서대로 **함께 의논해서** 정합니다.
- 확정된 내용은 담당 문서에만 반영하고, 같은 내용을 여러 문서에 반복하지 않습니다.
- 문제 정의서는 외부에서 작성해 들어온 입력 문서입니다. 온보딩에서는 이 문서를 수정하지 않고, 옆에 펼쳐 두고 맥락으로 참조합니다. (문제 정의서에는 핵심 문제·솔루션이 담겨 있습니다.)
- 이 온보딩에서 정하는 것은 협업 규약(②번)이며, 입력 문서(①번 문제 정의서)의 내용을 협업 규약 문서에 옮겨 적지 않습니다.

## 이 온보딩은 이렇게 진행합니다

- **팀장이 화면을 공유한 한 자리에서, 팀 전체가 함께** 진행합니다. 혼자 따로 돌리지 않습니다. (각자 돌리면 같은 결정 칸을 제각기 고쳐서 충돌납니다.)
- AI는 **한 번에 하나의 단계만** 안내합니다. 한 단계를 충분히 의논하고 넘어갑니다.
- **한 단계에서 묻는 횟수는 최대 2번입니다.** 질문을 잘게 쪼개 여러 번 주고받지 않습니다. 그 단계에서 정할 것·생각해 볼 것을 **선택지와 함께 한 번에 제시**하고, "팀이 의논해 정해 달라"고 합니다.
- **[이해하기] 시나리오는 답을 먼저 읊지 않되, 팀이 답하면 맞든 틀리든 곧바로 정답 기준과 이유를 알려 줍니다.** "거의 맞았어요, 나머지는?" 식으로 잘게 되묻지 않고, 틀린 답도 민망하지 않게 "왜 헷갈렸는지 → 실제 기준은 무엇인지 → 앞으로 어떻게 보면 되는지" 순서로 짚어 줍니다.
- **"정했으면 넘어감"이 기본입니다.** 정답을 찾는 자리가 아니라 팀 합의를 만드는 자리예요. 막히면 **기본값을 그대로 택하고** 넘어갑니다.
- **권장 시간(타이트하게):** 정하기 단계는 각 10분 안팎, 이해하기 단계는 각 5분 안팎.
- 팀원 대부분이 비개발자라는 점을 전제로, 용어는 일상 비유로 풀어서 설명합니다.
- 온보딩 중에는 문서 구조를 바꾸지 않고, 세부 기준만 조정합니다.
- 확정된 내용은 AI가 담당 문서의 `최종 결정`과 `변경 이력`에 반영하고, 팀은 그 결과를 함께 검토합니다.
- 한 단계가 끝날 때마다 "이 단계에서 우리가 알게 된 것"을 한 줄로 같이 확인하고 넘어갑니다.

## 온보딩이 끝나면 달라지는 것

- 팀원 모두가 **같은 협업 그림**(내 작업 공간에서 만들고 → 팀장이 완성본에 하나씩 합침)을 갖습니다. (브랜치·PR 같은 구체적인 절차는 작업하면서 익히고, 필요할 때 맨 뒤 **부록**을 참고합니다.)
- AI가 **어디까지 알아서 하고 어디서 멈추는지** 모두가 압니다.
- 데이터 이름이 제각각이 되지 않게 하는 **관문**이 무엇인지 압니다.
- 확정 사항이 각 문서의 `최종 결정`에 정리되고, 마지막에 사용자가 원하면 **통합 PRD 1.0** 한 부로 묶어 `docs`에 저장합니다.

## 단계는 두 종류입니다

온보딩은 앞쪽 `[정하기]` 단계 → 뒤쪽 `[이해하기]` 단계 순서로 진행합니다. AI는 단계 머리에 어떤 종류인지 표시합니다.

### 먼저, [정하기] 단계 — 팀이 함께 정하는 부분

- 팀마다 답이 달라지는 부분이라 실제로 논의가 필요합니다. 그래서 앞에서 먼저 끝냅니다.
- AI는 기본값을 설명한 뒤, 그 단계에서 정할 항목을 **선택지와 함께 한 번에** 제시하고 "팀이 의논해 정해 달라"고 합니다. 각 항목엔 "그대로 두면 안전한지, 바꾸면 무엇이 영향을 받는지"를 한 줄로 덧붙입니다.
- 팀이 답하면 AI가 담당 문서의 `최종 결정`과 `변경 이력`에 반영합니다. 마크다운 편집은 AI가 합니다.
- 해당 단계: 팀 구성, 디자인 방향(가장 많이 논의).

### 그다음, [이해하기] 단계 — 팀이 다 같이 보며 맞춰 보는 부분

- 합리적인 기본값을 따르며, **따로 정하거나 문서에 남길 건 없습니다 — 읽고 맞춰보기만** 합니다. 목적은 팀원 모두가 **"왜 이렇게 하는지"까지 같은 그림**을 갖는 것이에요. (순수 퀴즈가 아니라 "배우며 함께 맞춰 보는" 단계입니다.)
- 진행 방식: AI가 개념을 일상 비유로 풀어 설명 → 좀 더 자세한 설명과 흔한 오해 → **정답이 정해진 OX가 아니라, 팀이 같이 얘기해 볼 시나리오 질문** → 팀이 대화하며 맞춰 봄 → 핵심을 같이 확인하고 넘어갑니다.
- **AI 진행 규칙:** 시나리오는 **한 번에 제시**하고 "팀이 의논해 답해 달라"고 합니다. 팀이 답하면 **맞든 틀리든 곧바로 정답 기준과 이유를 알려 주고** 넘어갑니다. 정답은 먼저 말하지 않지만, 답변 뒤에는 숨기지 않습니다 — AI가 기준 문서(예: `AGENTS.md`)를 보고 **정답, 이유, 다음에 보는 기준**을 자연스럽게 짚어 줍니다. 답을 잘게 쪼개 되묻지 않습니다. (한 단계에서 묻는 횟수는 최대 2번.)
- 팀이 틀린 답을 해도 "아니요" 한마디로 끝내지 않습니다. 왜 그렇게 생각할 수 있는지 인정한 뒤, 실제 기준을 짧은 예시와 함께 알려 줍니다.
- 설명은 "외워야 할 규칙"이 아니라 "왜 그렇게 하면 우리가 편한지"를 이해시키는 데 초점을 둡니다.
- 해당 단계: AI 작동 규칙, 데이터 관문. (협업 안전장치와 배포 범위는 온보딩 흐름에서 다루지 않는 참고 문서로 둡니다.)
- 단계별 리듬: **3단계**(AI 작동 규칙)는 시나리오 질문으로, **4단계**(데이터 관문)는 짧은 **실습**으로 함께 맞춰 본 뒤 사전 맛보기를 출력하고 마무리합니다. AI는 각 단계 머리에서 어느 쪽인지 미리 알려 줍니다.

## 시작 요청 문구

수강생은 AI에게 아래처럼 요청합니다.

- 온보딩 시작!
- 이 레포를 처음 클론했습니다. 문제 정의서를 붙여넣었으니, 문서 기준을 지키면서 우리 팀 협업 온보딩을 시작해줘.

## 0단계: 입력 자료 확인

아래 두 가지를 대화창에 넣어 주세요.

1. 문제정의서 시트에서 출력한 **Markdown(md)**
2. Firebase 콘솔에서 복사한 **Web SDK 설정값**

AI는 Markdown에서 핵심 문제·솔루션을 읽어 정리하고, Firebase SDK 값은 `.env`에 저장합니다.

- 이 문서는 온보딩 내내 **참조만 하고 수정하지 않습니다.** (기획 문서라, 협업 규약과는 영역이 다릅니다.)
- Markdown에는 최소한 **핵심 문제와 솔루션 방향**이 담겨 있으면 됩니다.
- 항목이 빠졌거나 맥락이 부족하면 AI가 무엇이 더 필요한지 **짧게 한 번** 묻습니다.
- Firebase SDK 값은 문서에 기록하지 않고 `.env`에만 저장합니다.
- `.env`는 Git에 올리지 않으며, `.env.example`에는 값 없이 변수 이름만 남깁니다.

### 체크

- 입력 문서(문제정의서 시트에서 출력한 Markdown) 존재 확인
- Firebase Web SDK 설정값 존재 확인
- 핵심 문제·솔루션이 읽히는지 확인
- 입력 문서는 수정하지 않고 참조만 한다는 점 확인
- Firebase SDK 값은 `.env`에 저장하고 문서에는 남기지 않는다는 점 확인

## 1단계: 팀 구성 — [정하기]

기준 문서: `README.md`

### 쉽게 설명

- 프로젝트의 입구를 함께 확인하고, 팀에서 딱 하나만 정합니다: **누가 팀장인지.**
- 팀장이 하는 일은 **PR 최종 병합과 배포** 단 두 가지입니다. (여기서 'PR 병합'은 *팀원이 만든 작업을 완성본에 합치는 마지막 확인 절차*예요. 자세한 개념은 맨 뒤 부록에 있으니 지금은 "팀장이 완성본 문을 지킨다" 정도로만 이해하면 됩니다.) 즉 "완성본(main)에 무엇을 언제 들일지"를 정리하는 역할이에요.
- 왜 한 명이 맡냐면 — 여러 명이 동시에 완성본에 손대면 작업이 뒤섞여 깨지기 쉽습니다. **들어오는 문을 한 명이 지키면** 순서가 꼬이지 않습니다.
- 팀장은 **코딩을 제일 잘하는 사람일 필요가 없습니다.** 흐름을 차분히 정리할 사람이면 됩니다.

### 팀장이 하는 일 / 하지 않는 일

- ✅ 하는 일: PR을 하나씩 순서대로 병합, 배포(Hosting) 실행, 충돌이 났을 때 AI 해결안을 보고 최종 확인.
- ❌ 하지 않는 일: 팀원 코드를 대신 짜 주기, 모든 결정을 혼자 내리기. (결정은 팀이 같이, 반영은 AI가)

### 작업 브랜치는 어떻게 쓰나 (참고)

- 각자 작업은 `student/이름/작업명` 형식의 개인 브랜치에서 진행합니다. (예: `student/gildong/candidate-list`)
- 이름은 영문 소문자 등 Git에서 안전한 표기를 씁니다. 따로 외울 필요는 없고, 작업을 시작할 때 정하면 됩니다.
- 자세한 브랜치·PR 흐름은 맨 뒤 **부록**을 참고합니다.

### 🗣️ 함께 정할 것

- **팀장은 누구로 할까요?** 이름만 정하면 됩니다. (이 단계는 이 한 가지만 정하면 끝입니다.)

> 🛟 그대로 두면: 팀장이 없으면 누가 완성본에 합칠지가 안 정해져 나중에 서로 main에 손대다 충돌이 납니다. 그래서 한 명은 꼭 정하고 넘어갑니다.

### 체크

- 팀장 확정

> 이 단계에서 알게 된 것: **팀장은 '완성본에 들이는 문'을 지키는 한 사람. 코딩 실력과 무관.**

## 2단계: 디자인 방향 — [정하기 · 가장 많이 논의]

기준 문서: `docs/design-system.md`

### 쉽게 설명

- 팀마다 다른 AI로 화면을 만들어도 **한 서비스처럼 보이게** 하는 최소 디자인 기준입니다.
- 비개발자가 가장 의견을 내기 좋은 단계입니다. 정답이 없으니 **각자 취향을 편하게 꺼내 놓는 자리**예요. 많이 말할수록 결과가 팀 취향에 가까워집니다.
- 기본 골격(파란색 계열, 카드·표·버튼 기준)은 유지하되, 그 위의 분위기와 디테일을 팀이 정합니다.

### 왜 디자인 기준을 미리 맞추나

- 각자 다른 AI에게 화면을 맡기면, 같은 버튼인데 누구는 둥글고 누구는 각지고, 색도 제각각이 됩니다. 합쳐 놓으면 **여러 사이트를 섞어 놓은 느낌**이 나요.
- 그래서 "색/버튼/카드/표/여백" 같은 **최소한만 미리 맞춰** 두면, 세부는 달라도 한 서비스처럼 보입니다.
- 단, 이건 **개발을 막는 사전 승인 장치가 아닙니다.** 일단 만들고, 톤이 어긋난 화면만 나중에 "디자인 시스템 기준으로 정리해줘" 한 번에 맞추는 방식이에요.

### 이미 정해진 기본값 (바꾸고 싶을 때만 의논)

- 분위기: 조용하고 실용적인 업무 도구 UI (장식보다 정보·사용성 우선)
- 메인 색상: 파란색 계열 / 배경: 흰색·옅은 회색 / 텍스트: 진한 회색
- 버튼: 기본·보조·위험 3종만 / 카드: 목록·정보 묶음 / 표: 데이터 비교·관리
- 모바일에서 글자·버튼이 잘리지 않게 유지

### 공통 컴포넌트는 "초기 베이스 + 나중에 통합"

- 프로젝트 초기에 버튼·입력창·카드·표 같은 **베이스 공통 컴포넌트 한 벌**을 만들어, 모두 같은 데서 출발합니다.
- 개발 중에 새로 필요한 건 **각자 자기 플로우에서 그냥 만듭니다(중복 OK).** 실시간으로 공용 코드를 같이 고치려 하지 않습니다 — 병렬 개발에선 그게 오히려 충돌을 만들어요.
- 반복되는 게 쌓이면 **나중에 리팩토링에서 공통으로 합칩니다.** ("일단 개별 → 나중에 통합")

### 🗣️ 함께 정할 것 (~10분 · 아래를 의논해 한 번에 정하기)

> 정답이 없는 자리예요. 아래 5가지를 **팀이 한 번에 의논해** 정해 주세요. AI는 항목마다 따로 묻지 않고, 팀이 정한 결과를 한꺼번에 받아 적습니다. 막히는 항목은 기본값(파란색·미니멀)으로 두면 됩니다.

1. **레퍼런스 — 1인 1개만.** 각자 평소 좋아하는 서비스/사이트를 *하나씩만* 떠올려 "이 느낌 좋다 / 이건 싫다"를 한마디로. (URL·이미지를 주면 그 톤을 기준으로 잡습니다. 여러 개 늘어놓지 않습니다.)
2. **분위기 — 셋 중 하나로.** ① 깔끔·미니멀 ② 부드럽고 친근 ③ 단단한 업무 도구 → 팀이 하나를 고릅니다. (못 고르면 ①)
3. **메인 색상 — 한 줄.** 선호 색이 있나요? 없으면 **파란색 유지**로 확정.
4. **선호/비선호 요소 — 각 1개씩만.** 꼭 넣고 싶은 것 1개, 피하고 싶은 것 1개. (예: 둥근 모서리 선호 / 과한 그림자 비선호)
5. **운영 강도 — 확인만.** "최소 기준만 맞추고 세부는 AI가 보완, 약한 불일치는 후속 정리"로 갑니다. 그대로 두면 빠르고, 깐깐할수록 느려집니다. → 이대로 OK?

> **합의 한 줄로 마무리:** "우리 팀 디자인 = [분위기] + [색] + [꼭 넣을 것/뺄 것]." AI가 이 한 줄을 `최종 결정`에 적습니다.

### 흔한 오해

- ❌ "디자인은 완벽히 통일돼야 PR을 합칠 수 있다" → ✅ 아니요. 약한 불일치는 후속 정리 대상입니다. 단 **기본 색상/버튼/카드/표**처럼 골격은 크게 벗어나면 안 됩니다.

### 체크

- 참고 레퍼런스 수집
- 분위기 방향 확정
- 메인 색상 확정
- 선호·비선호 요소 정리
- 디자인 운영 강도 확정 (최소 기준 + 후속 정리)
- 공통 컴포넌트 운영 확인 (초기 베이스 한 벌 + 개발 중 중복 감수 + 후속 리팩토링 통합)

> 이 단계에서 알게 된 것: **최소 기준만 공유하면 다른 AI로 만들어도 한 서비스처럼 보인다.**

## 3단계: AI 작동 규칙 — [이해하기]

기준 문서: `AGENTS.md`

### 쉽게 설명

- AI는 아주 유능한 팀원이지만, **알아서 다 해버리면 오히려 위험**할 때가 있습니다. 그래서 "어디까지 스스로 하고, 어디부터 멈추고 물어볼지" 선을 정해 둡니다.
- 비유하면 — 심부름은 알아서 하지만, **집 구조를 바꾸는 일은 가족에게 먼저 물어보는** 것과 같습니다.

### 어디까지 알아서 / 어디서 멈추나

- **AI가 알아서 하는 것:** 요청받은 코드 수정, 문서 수정, 실행·빌드·린트 확인, 커밋, PR 생성.
- **AI가 멈추고 먼저 물어보는 것 (되돌리기 어렵거나 팀 전체에 영향):**
  - 파일 삭제
  - 코드 대규모 이동
  - 새 라이브러리(의존성) 추가
  - 공통 구조 변경 — 여기서 "공통 구조"란 **① SPA 구조 ② 역할 기반 화면 분기**를 말합니다.
- AI는 막무가내로 멈추는 게 아니라, **"왜 필요한지 설명하고 → 물어보고 → 진행"** 합니다.

### AI가 물어보면 우리는 어떻게 답하나

- AI가 "이거 지워도 될까요?" "이 라이브러리 추가할까요?"라고 하면, **이유를 보고** 판단합니다.
- 잘 모르겠으면 "왜 필요한지 한 줄로 설명해줘"라고 되물으면 됩니다. 그게 정상적인 협업이에요.

### 흔한 오해

- ❌ "AI니까 알아서 다 잘 처리하겠지" → ✅ 평범한 작업은 그렇지만, **위험한 작업은 일부러 멈추도록** 약속해 둔 겁니다. 멈추는 건 AI가 일을 못 해서가 아니라 안전장치예요.

### 🗣️ 함께 얘기해 볼 것 (~5분 · 의논해서 한 번에)

목표: 목록을 외우는 게 아니라 **"언제 정신 차리고 봐야 하는지" 감**을 잡는 것. 아래를 팀이 **의논해서 한 번에** 답해 보세요. AI는 답을 먼저 말하지 않고, 팀이 답하면 곧바로 이유를 알려 줍니다.

**골라보기 — 다음 중 AI가 *멈추고 물어봐야 하는 것*은? (여러 개)**

- (가) "버튼 색을 파란색으로 바꿔줘"
- (나) "안 쓰는 옛날 폴더 통째로 지워줘"
- (다) "이 기능 만들려면 새 라이브러리가 필요해요"
- (라) "지원자 목록에 정렬 기능 추가해줘"
- (마) "역할별 화면 분기 방식을 통째로 바꿀게요"

**함께 떠올려 볼 점** (골라보기와 같이 얘기해 보세요):
- ⓐ AI가 "이 라이브러리 추가할까요?"라고 물으면 우리 팀은 뭐라고 답할까?
- ⓑ 매번 "응 그냥 해"만 누르면 팀 전체에 무슨 일이 생길까?
- 앞으로 우리가 만들 화면·기능을 떠올려 같은 기준으로 비춰 보기 — "그 작업 중 AI가 멈춰 물어볼 대목이 있을까?" (대부분 "없음"이 정상)

팀이 답하면 AI는 아래 순서로 짚어 줍니다.

1. 팀 답이 맞았는지 먼저 말합니다.
2. 정답 기준을 알려 줍니다. 이 예시의 기준 답은 **(나), (다), (마)** 입니다.
3. 왜 (나) 삭제, (다) 의존성 추가, (마) 공통 구조 변경은 멈춰야 하는지 설명합니다.
4. 왜 (가) 작은 UI 수정, (라) 일반 기능 추가는 보통 바로 진행해도 되는지 설명합니다.
5. 마지막에 "위험하거나 되돌리기 어려운 일은 먼저 멈춘다"는 한 줄로 정리합니다.

> 이 단계에서 알게 된 것: **AI가 멈추고 물어보는 건 안전장치다. 위험·되돌리기 어려운 일(삭제·대규모 이동·의존성 추가·공통 구조 변경)은 먼저 멈춘다. 우리는 이유를 읽고 판단한다 — 무조건 승인도, 무조건 거부도 아니다.**

## 4단계: 데이터 관문 — [이해하기]

기준 문서: `docs/data.md`

### 쉽게 설명

식당에서 누구는 "coke", 누구는 "콜라"라고 적으면 — 사람은 알아듣지만 **프로그램은 둘을 완전히 다른 것으로 봅니다.** 데이터 이름도 똑같아요. 같은 걸 다르게 부르면 합칠 때 안 맞아 전부 고쳐야 해요.

그래서 데이터를 새로 만들 때는 만들기 전에 **"메뉴판(공통 이름 사전)에 이미 있나?"부터 봅니다.** 있으면 그대로 쓰고, 없을 때만 사전에 등록하고 써요. 이 문을 **관문**이라고 부릅니다 — AI가 매번 자동으로 지키니 팀원이 외울 필요는 없습니다.

### 🗣️ 함께 해 볼 것 — 실습 (~5분)

**지원자의 "나이"를 저장한다고 해봅시다. 가장 먼저 해야 할 일은?** (팀이 의논해서 골라보세요)

1. AI에게 이 단어가 사전에 이미 있는지 확인하라고 한다
2. 그냥 추가한다

팀이 답하면 AI는 바로 기준을 알려 줍니다.

- 정답은 **1번**입니다.
- 2번을 골랐다면, `age`, `candidateAge`, `birthYear`처럼 같은 의미가 여러 이름으로 갈라질 수 있음을 설명합니다.
- 이후 흐름은 **사전에 있는지 확인 → 있으면 재사용 → 없으면 사전에 등록 후 사용**이라고 정리합니다.

### 📖 사전 맛보기 (읽고 "이런 게 있구나"만)

AI는 실습 답변에 이유를 알려 준 뒤, 아래 **공통 이름 사전(`docs/data.md`) 맛보기**를 반드시 출력합니다. 나중에 데이터가 필요할 때 "아, 사전에 있었지?" 하고 여기부터 찾아보게요.

**이 프로젝트 사전에 이미 들어 있는 것 (대략):**

- 컬렉션 **13개** · 공통 필드 **20여 개** · 역할 값 **3개(고정)** · 상태값 **11개**

**맛보기 — 이런 이름들이 있어요:**

| 종류 | 예시 |
| --- | --- |
| 컬렉션(표) | `users`(유저) · `applications`(지원서) · `interviews`(면접) · `evaluations`(평가) · `jobPostings`(채용공고) |
| 공통 필드(칸) | `id` · `userId` · `status`(상태) · `title`(제목) · `createdAt`(생성시각) · `updatedAt`(수정시각) |
| 역할 값(고정) | `candidate`(지원자) · `recruiter`(채용담당자) · `interviewer`(면접관) |
| 상태값(정해진 값만) | `pending`(대기) · `confirmed`(확정) · `completed`(완료) · `cancelled`(취소) · `draft`(초안) |

**공통 필드 맛보기 — 실제로 이런 칸 이름을 씁니다:**

| 필드 | 뜻 | 언제 쓰나 |
| --- | --- | --- |
| `id` | 문서 또는 항목 식별자 | 모든 데이터의 기본 이름표 |
| `userId` | 유저 식별자 | 로그인한 사람이나 특정 유저를 연결할 때 |
| `candidateId` | 지원자 식별자 | 지원자 프로필, 지원서, 면접을 지원자와 연결할 때 |
| `recruiterId` | 채용담당자 식별자 | 채용공고나 일정 조율 담당자를 연결할 때 |
| `interviewerId` | 면접관 식별자 | 면접 일정이나 평가를 면접관과 연결할 때 |
| `companyId` | 기업 식별자 | 유저, 공고, 지원서를 특정 기업과 연결할 때 |
| `jobPostingId` | 채용공고 식별자 | 지원서나 면접이 어떤 공고의 것인지 연결할 때 |
| `applicationId` | 지원서 식별자 | 면접, 평가가 어떤 지원서에서 이어졌는지 연결할 때 |
| `interviewId` | 면접 식별자 | 질문, 평가, 일정이 어떤 면접에 속하는지 연결할 때 |
| `status` | 상태값 | 대기, 확정, 완료, 취소 같은 진행 상태를 표시할 때 |
| `title` | 제목 | 공고명, 질문 세트명, 화면에 보이는 이름이 필요할 때 |
| `description` | 설명 | 상세 설명이나 보조 내용을 저장할 때 |
| `startedAt` | 시작 시각 | 면접, 가능 시간, 진행 시간이 시작되는 때 |
| `endedAt` | 종료 시각 | 면접, 가능 시간, 진행 시간이 끝나는 때 |
| `scheduledAt` | 확정 시각 | 일정이 확정된 시간을 저장할 때 |
| `source` | 생성 또는 판단 근거 | AI 추천, 사용자 입력 등 데이터가 나온 근거를 남길 때 |
| `createdAt` | 생성 시각 | 데이터가 처음 만들어진 시간 |
| `updatedAt` | 수정 시각 | 데이터가 마지막으로 바뀐 시간 |

> 외울 필요 없습니다. **"이런 사전이 있다"는 것만** 기억하세요 — 데이터가 필요할 때 AI가 여기부터 찾아 줍니다.

> 이 단계에서 알게 된 것: **새 데이터는 항상 공통 이름 사전을 먼저 거친다(있으면 재사용·없으면 등록). 역할·상태값은 정해진 것만. 이름을 미리 맞춰 두면 합칠 때 안 깨진다.**

이해하기 단계에서 함께 맞춰 본 것:

(이해하기 단계(3·4)는 따로 정하거나 문서에 남길 게 없어요 — 읽고 맞춰보기만 합니다. 배포 범위는 온보딩에서 묻지 않고 실제 배포가 필요할 때 `docs/firebase-guide.md`를 확인합니다.)
(협업 안전장치(브랜치·PR·병합, 독립 구현 원칙)는 온보딩 흐름에서 따로 다루지 않는 **부록**입니다. `docs/project-rules.md`에 이미 반영돼 있어 기본값으로 적용되며, 필요할 때 부록을 참고합니다.)

- AI가 반영할 문서 목록 확인

### 문서 반영 기준

- 온보딩에서 확정한 내용은 AI가 담당 문서의 `최종 결정`에 반영합니다.
- 기준이 바뀐 이유는 AI가 `변경 이력`에 남깁니다.
- 아직 정하지 못한 내용은 AI가 담당 문서의 `논의할 항목`에 남깁니다.
- 사람은 반영 결과를 검토하고, 입력 문서(문제 정의서)는 수정하지 않습니다. Firebase SDK 값은 `.env` 저장 여부만 확인합니다.

### PRD 1.0 생성 확인

협업 규약이 확정되면 AI는 바로 PRD를 만들지 않고 먼저 묻습니다.

> PRD 1.0을 생성하시겠습니까?

사용자가 동의하면 AI가 통합 PRD 1.0을 생성하고, 생성과 동시에 `docs/prd-1.0.md`에 저장합니다.

- 구성: ① 문제 정의서(핵심 문제·솔루션) ② 확정된 협업 규약 결정 요약.
- ①번은 0단계 입력 Markdown의 핵심을 의미 그대로 싣고, ②번은 온보딩에서 실제로 정한 내용과 각 담당 문서의 `최종 결정`만 모아 정리합니다.
- AI는 결정을 임의로 바꾸거나 살을 붙이지 않습니다.
- PRD 본문에는 Firebase SDK 값 같은 비밀값을 적지 않습니다.
- 정하지 못한 항목은 PRD 안에 `논의할 항목`으로 남겨 둡니다.

### 다음 단계

PRD 1.0을 생성했다면 각 수강생이 개인 브랜치를 만들고, 자기 작업에 맞는 담당 문서만 확인해 AI에게 요청합니다.

- 🏠 **브랜치** = 각자 작업하는 "내 방". 내가 무엇을 고쳐도 남의 작업이나 완성본에 영향이 없습니다.
- 📋 **PR** = 내 방에서 만든 걸 "검사대에 올려 두는 것". 합치기 전에 한 번 점검하는 자리입니다.
- 🏛️ **main** = 완성본이 모이는 "본관". 아무나 못 들어가고, **팀장이 하나씩** 들입니다.

### 실제로 한 바퀴 도는 모습

팀원 한 명이 AI에게 **"지원자 목록 화면 만들어줘"** 라고 하면:

1. AI가 그 팀원의 **작업 방(브랜치)** 을 만들고 (`student/이름/작업명`)
2. 그 안에서 코드를 짜고
3. **검사대(PR)** 에 올린 뒤
4. **스스로 리뷰**해서 "에러 없는지, 공통 구조 안 깨지는지" 확인하고
5. **팀장이** 확인하고 **본관(main)** 에 들입니다.
6. 그러면 다른 팀원들은 최신 본관을 **받아와서(pull)** 그 작업을 같이 보게 됩니다.

→ 핵심은 **"누구도 완성본(main)에 직접 손대지 않는다"** 입니다. 항상 각자 방 → 검사대 → 팀장 순서예요.

### 새로 나온 단어 정리 (비개발자용)

- **push(올리기):** 내 방에서 만든 걸 인터넷의 우리 레포로 올리는 것.
- **pull(받아오기):** 남이 올린 최신 내용을 내 컴퓨터로 받아오는 것.
- **PR(Pull Request):** "내 작업을 본관에 합쳐 주세요"라고 요청하는 검사대.
- **병합(merge):** 검사대를 통과한 작업을 본관에 합치는 것. (팀장만)
- **충돌(conflict):** 두 사람이 같은 곳을 다르게 고쳐서, 어느 쪽을 따를지 정해야 하는 상태.

### 우리 팀의 충돌 예방 원칙

- 우리 팀은 **각 유저 플로우를 완전히 독립적으로 구현**합니다.
- 비슷한 코드가 좀 중복되더라도, **다른 사람 작업에 기대지 않게** 만듭니다.
- 이유: 서로 무엇을 작업 중인지 정확히 몰라도 **병합할 때 부딪히지 않게** 하려는 거예요. 충돌을 "해결"하기보다 아예 "안 생기게" 하는 전략입니다.
- 단, **데이터 이름(공통 이름 사전)은 예외로 함께 공유**합니다. 데이터 이름은 오히려 같아야 화면끼리 맞아요. → **구현 코드는 독립, 데이터 이름은 공유.**

### PR을 합칠 때 AI가 보는 것 / 팀장이 보는 것

- **AI 리뷰가 확인:** 실행·빌드 에러 없는지, 병합 후 깨질 충돌 없는지, SPA·역할 분기 같은 공통 구조를 훼손하지 않는지, 문서 기준을 어기지 않는지.
- **팀장이 확인:** AI 리뷰 결과를 보고 **하나씩 순서대로** 병합. 여러 개를 한꺼번에 합치지 않습니다.
- **병합 뒤:** 그 작업 브랜치는 정리(삭제)하고, 다음 브랜치는 최신 본관 기준으로 이어 갑니다.

### 흔한 오해

- ❌ "내 작업 다 됐으니 그냥 main에 바로 넣자" → ✅ 안 됩니다. 무조건 브랜치 → PR → 팀장 병합.
- ❌ "PR은 긴 회의용 문서다" → ✅ 아니요. **본관 들어가기 전 가벼운 체크포인트**입니다.
- ❌ "PR이 여러 개면 한 번에 합치면 빠르다" → ✅ 한꺼번에 합치면 무엇이 문제를 냈는지 못 찾습니다. 하나씩.

### 핵심 한 줄

> **내 브랜치에서 작업 → AI가 PR로 올려 리뷰 → 팀장이 하나씩 순서대로 병합.** 완성본에는 직접 손대지 않는다. 충돌은 "해결"보다 "예방"(독립 구현 + 공통 영역만 조심).
